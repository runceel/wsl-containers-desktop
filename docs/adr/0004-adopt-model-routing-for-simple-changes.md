# 0004. 単純な修正向けのモデルルーティング方針を採用する

## Status

Accepted

## Context

- Copilotエージェントで開発する際、モデルによってトークンコスト・応答速度・推論品質が異なる。
- タイポ修正、フォーマット調整、単純なリネームなど「判断を伴わない機械的な修正」にまで
  高性能・高コストなモデルを使うのはコスト効率が悪い。
- 一方で、設計判断・アーキテクチャレビュー・ラバーダック・ADR作成・TDDの各フェーズ
  （ADR-0002）は、判断の質が成果物の品質に直結するため、モデルの性能を落とすべきではない。

## Decision

- 「設計判断を伴わない機械的な修正」に限り、`model: mai-code-1-flash-picker` を明示指定した
  専用custom agent `.github/agents/quick-fix.agent.md` を使う。
  - 対象例: タイポ修正、フォーマット/インデント調整、変数・メソッドのリネーム（挙動不変）、
    デッドコード削除、コメント追記、明らかなnull参照ガード追加など。
  - 対象外: 新機能実装、バグ修正（挙動の変更を伴うもの）、設計変更、
    ADR作成、TDDの各フェーズ（Red/Green/Refactor）、ラバーダックレビュー。
- 上記「対象外」に該当する作業は、通常の（高性能な）モデルを使う既存の開発フローに従う。
- 迷った場合は「対象外」側として扱い、通常のフローに従う（コスト削減よりも品質を優先する）。

## Consequences

- 機械的な小修正のトークンコストを削減できる。
- モデル選択の基準が明文化されることで、Copilotエージェント・人間コントリビューター双方が
  一貫した判断をできるようになる。
- 判断を誤って「対象外」の作業を`quick-fix` agentに任せてしまうリスクがあるため、
  `quick-fix.agent.md` 自体に対象外条件を明記し、該当しそうな場合は通常フローへの
  切り替えを促す記述を入れる。
