# 0002. 厳密なTDD(Red-Green-Refactor)を採用する

## Status

Accepted

## Context

- WSL Containers Desktop は、コンテナのライフサイクル管理・状態表示など、
  正確性が求められるロジック（Docker Desktop相当の機能）を多数含む見込み。
- Copilotエージェントに実装を任せる場面が多くなるため、「動くように見えるが仕様を満たしていない」
  実装が紛れ込むリスクを機械的に防ぎたい。
- テストを後から書く運用では、実装に都合の良いテストになりがちで、仕様の見落としに気づきにくい。

## Decision

- すべての機能実装は、厳密な **Red → Green → Refactor** のTDDサイクルで行う。
  - **Red**: 仕様を表す失敗するテストのみを書く。プロダクションコードは変更しない。
  - **Green**: 直前に追加した失敗テストを通す**最小限**の実装のみを行う。
  - **Refactor**: 外部から見た挙動を変えずに設計を改善する。各変更ごとにテストを再実行する。
- 各フェーズは専用のCopilot custom agentで実行する:
  `.github/agents/tdd-red.agent.md` / `tdd-green.agent.md` / `tdd-refactor.agent.md`。
- テスト作成の前に、テスト内容自体をラバーダック（`rubber-duck` agent）でレビューする
  （開発フロー全体は [`AGENTS.md`](../../AGENTS.md) を参照）。
- 単体テストフレームワークの選定は ADR-0003 で扱う。

## Consequences

- 各コミット単位で「テストが仕様を表現している」ことが保証されやすくなる。
- 実装速度は多少落ちるが、手戻り・仕様漏れによる手戻りコストを削減できる。
- 単純な機械的修正（タイポ修正等）にまでこのサイクルを強制すると非効率なため、
  そのような変更は ADR-0004 のモデルルーティング方針に基づき `quick-fix` agent を使い、
  TDDサイクルの厳密な適用対象は「振る舞いに影響する実装」に限定する。
