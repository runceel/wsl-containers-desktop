# 0011. 設定変更時にWSLを自動再起動せず案内のみ行う

## Status

Accepted

## Context

- [Issue #7 [0007] 設定画面](https://github.com/runceel/wsl-containers-desktop/issues/7)
  の実装にあたり、リソース制限（メモリ量・論理プロセッサ数）を `%USERPROFILE%\.wslconfig` の
  `[wsl2]` セクションに保存・リセットする機能を追加する。
- [`docs/reference/wsl-containers-platform.md`](../reference/wsl-containers-platform.md) および
  Microsoft Learnで確認した事実として、`.wslconfig` の `[wsl2]` 設定（`memory`/`processors`/`swap`等）は
  **WSLの再起動（`wsl --shutdown`）を行わないと反映されない**。
- `wsl --shutdown` はWSL 2ユーティリティVMを停止するため、**実行中のすべてのディストリビューション・
  コンテナ・プロセスを巻き添えに終了させる**。本アプリはWSL Containers上のコンテナを管理する
  ツールであり、ユーザーが把握しないうちに実行中コンテナを停止させることは重大なデータ損失・
  作業中断のリスクになる。
- Issue #7の受け入れ基準は「保存後、設定を反映するにはWSLの再起動が必要である旨がユーザーに
  伝わること」を要求しており、自動再起動そのものは要求していない。

## Decision

設定（リソース制限）の保存・リセット時に、アプリは **`wsl --shutdown` を自動実行しない**。
代わりに、保存・リセットが成功したら **「反映にはWSLの再起動（`wsl --shutdown`）が必要」である旨を
ユーザーに案内する**（Presentation層のステータスメッセージ）。

- この方針は当面「リソース制限」設定に適用する。将来、再起動を伴う他の設定を追加する場合も
  既定の方針として踏襲する。
- 再起動の実行タイミングはユーザー自身の判断に委ねる（アプリ内から再起動する導線を将来設ける
  場合は、実行中コンテナへの影響を明示する確認UIを伴うこととし、その時点で別途判断する）。

## Consequences

- 実行中コンテナをユーザーの意図しないタイミングで停止させるリスクを排除できる。
- 保存しただけでは設定が即座に反映されないため、「保存したのに変わらない」という誤解を招かないよう、
  再起動が必要である旨のメッセージを明確に提示することが必須となる（受け入れ基準を満たす前提条件）。
- Infrastructure層に `wsl --shutdown` を実行する処理を持たないため、設定保存の副作用が
  ファイル書き込みに限定され、テスト・実装が単純になる。
