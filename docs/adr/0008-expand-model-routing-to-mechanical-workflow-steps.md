# 0008. モデルルーティング方針をワークフロー内の機械的ステップへ拡張する

## Status

Accepted

## Context

- ADR-0004により、「設計判断を伴わない機械的な修正」専用の`quick-fix` agent
  （`mai-code-1-flash-picker`固定）を導入した。ADR-0004では「対象外」として
  ADR作成・TDDの各フェーズ・ラバーダックレビューを一律に除外していた。
- しかし実際の開発フローを見直すと、これらのプロセスの中にも、判断がすでに
  完了した後の**純粋な機械的作業**が含まれている。
  - ADR/design docの一覧表への追記（内容が確定した後、目次に1行足すだけの作業）。
  - ラバーダックレビューで得られた指摘のうち、タイポ・フォーマットレベルの軽微なもの。
  - 明らかに不要なusing/importの整理、既存パターンをそのままコピーする定型的な
    ボイラープレート追加。
  - TDD Redフェーズ: ADR-0002により、Redフェーズで書くテストは事前にラバーダックで
    レビュー済みの仕様を反映したものであることが既に前提となっている。この前提を
    「具体的な入出力・アサーション値・エッジケースまで確定済み」の水準まで強化すれば、
    Redフェーズの実作業は「確定済み仕様のコードへの書き起こし＋実行してRedを確認する」
    という機械的作業に近づく。
- 一方で、これらを一律にFlashへ倒すと、ADR-0004が懸念した「判断を誤って対象外の作業を
  低コストモデルに任せてしまうリスク」が再燃する。特にTDD Redフェーズは、テストの質が
  Green/Refactor以降の正しさの土台になるため、影響範囲が大きい。
- 検討した代替案:
  - TDD Redフェーズを常にFlash対象外のままにする案 → 却下。フェーズ3（テスト作成）の
    完了条件を強化すれば、Redフェーズの残作業は十分に機械的になるため、コスト削減の
    機会を逃す。
  - TDD Redフェーズを無条件でFlashに切り替える案 → 却下。仕様がまだ抽象的な場合や、
    コンパイルのために新しい設計判断（インターフェース配置等）が必要になる場合に、
    質の低下リスクが残る。

## Decision

- ADR-0004の対象を維持しつつ、以下の機械的ステップに限定して`quick-fix` agent
  （`mai-code-1-flash-picker`）の利用範囲を拡張する。
  1. `docs/adr/README.md` / `docs/design/README.md` / `docs/reference/README.md` の
     一覧表への追記（本文の執筆・内容判断が完了した後の目次反映のみ）。
  2. ラバーダックレビューで得られた指摘のうち、タイポ・フォーマット・軽微な文言修正
     レベルのものの反映（設計内容そのものの変更を伴う指摘は対象外のまま）。
  3. `quick-fix` agentの「対象」例に、明らかに不要なusing/import整理、既存パターンを
     そのままコピーする定型的なボイラープレート追加を追加する。
- TDD Redフェーズについては、以下の条件を**両方**満たす場合に限り、`tdd-red` agentの
  既定モデルを`mai-code-1-flash-picker`とする。
  1. 「テスト作成」フェーズ（フェーズ3）の完了条件を強化し、ラバーダックレビュー通過には
     具体的な入出力・アサーション値・エッジケース表まで確定していることを必須とする。
  2. `tdd-red` agentは開始時に、上記が確定済みかを確認する。確定していない場合は
     Flashで進めず、テスト作成フェーズに差し戻す。また、コンパイルのために新しい
     インターフェース設計判断（どの層に何を追加するか、シグネチャの決定等）が
     必要になった場合は、その場で通常モデルでの対応にエスカレーションする
     （`quick-fix` agentと同様の「迷ったら対象外」原則を適用する）。
- 上記以外（新機能実装、挙動を変えるバグ修正、設計・アーキテクチャ変更、ADR本文/design doc
  本文の執筆、TDD Green/Refactorフェーズ、ラバーダックレビューにおける設計判断そのもの）は
  引き続き通常モデルを使う既存の開発フローに従う。

## Consequences

- 開発フロー内でFlashを活用できる範囲が広がり、トークンコストをさらに削減できる。
- TDD Redフェーズのコスト削減は、フェーズ3（テスト作成）の完了条件強化とセットで
  初めて成立する。フェーズ3が疎かになると、Redフェーズの前提が崩れ品質低下リスクが
  再発するため、`feature-workflow` skillの完了条件（DoD）を必ず更新して運用する。
- `tdd-red` agentにエスカレーション条件（前提未確定・新規設計判断発生時は通常モデルへ）を
  明記する必要がある。これにより判断を誤るリスクを抑える。
- 影響を受けるファイル: [`.github/agents/quick-fix.agent.md`](../../.github/agents/quick-fix.agent.md)、
  [`.github/agents/tdd-red.agent.md`](../../.github/agents/tdd-red.agent.md)、
  [`.github/skills/feature-workflow/SKILL.md`](../../.github/skills/feature-workflow/SKILL.md)、
  [`.github/instructions/tests.instructions.md`](../../.github/instructions/tests.instructions.md)。
- ADR-0004は本ADRにより置き換えられるものではなく、引き続き有効（`Status: Accepted`）。
  本ADRはADR-0004の枠組みを維持したまま、適用範囲を拡張するものである。
